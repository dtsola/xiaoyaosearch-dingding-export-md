## 1. 总体技术架构

### 架构模式
- **架构类型**：浏览器扩展（Browser Extension）
- **部署方式**：本地扩展（用户自主安装）
- **核心特性**：
  - 无需后端服务器，纯前端实现
  - 基于 File System Access API 实现本地文件操作
  - 利用钉钉网页版 Cookie 进行认证
  - 支持多种文档格式转换
- **模块化设计**：
  - 后台脚本模块（background.js）
  - 内容脚本模块（content.js）
  - UI 组件模块（Baby 框架 + 业务组件）
  - API 封装模块（api.js）

## 2. 前端技术选型

### 2.1 核心框架

#### **Baby 框架（自研响应式框架）**
- **选择原因**：
  - 继承自老项目 ding-doc-downloader，代码成熟稳定
  - 轻量级（约 400 行），满足组件化需求
  - 与现有代码库完全兼容，降低迁移成本
- **核心优势**：
  - 基于 `Object.defineProperty` 的响应式数据绑定
  - 组件化开发模式，支持 props 和事件传递
  - 生命周期钩子（created、mounted）
  - watch 机制，监听数据变化
- **替代方案对比**：
  - **Vue.js**：生态完善，但增加项目体积，需要重构现有组件
  - **React**：生态强大，但 JSX 语法需要构建工具支持，学习成本高
  - **原生 JS**：无框架依赖，但组件化开发困难，代码可维护性差

#### **Vite v8**
- **选择原因**：现代构建工具，开发体验好
- **核心优势**：
  - 极快的冷启动速度
  - 即时的模块热更新（HMR）
  - 原生支持 ES 模块
  - 丰富的插件生态

### 2.2 UI 组件库

#### **Tailwind CSS v4 + DaisyUI v5**
- **选择原因**：
  - Tailwind CSS 提供原子化样式，便于定制 Apple 风格
  - DaisyUI 提供高质量组件，基于 Tailwind 构建
  - 老项目已使用，保持技术栈一致
- **核心优势**：
  - 高度可定制，支持自定义主题
  - 生产环境自动清除未使用的样式
  - 响应式设计开箱即用
  - 暗色模式支持
- **替代方案对比**：
  - **Ant Design**：组件丰富，但定制困难，体积较大
  - **Element Plus**：Vue 生态组件库，不适用于 Baby 框架
  - **Bootstrap**：传统 UI 框架，定制性较差

### 2.3 状态管理

#### **Baby 框架内置响应式系统**
- **选择原因**：Baby 框架自带响应式能力，无需额外引入状态管理库
- **核心优势**：
  - 轻量级，与框架深度集成
  - 基于 getter/setter 的数据劫持
  - 组件间通信通过事件系统
- **使用场景**：
  - 文档列表状态
  - 下载进度状态
  - 用户配置状态

### 2.4 网络通信

#### **自定义 httpRequest（基于 XHR）**
- **选择原因**：
  - XHR 支持上传/下载进度监控
  - 更好的跨域兼容性
  - 可精细控制请求头（钉钉 API 需要特殊认证头）
- **核心优势**：
  - 进度回调支持（onProgress）
  - 自定义请求头
  - 错误处理完善

### 2.5 其他前端库

#### **PostCSS v8**
- **选择原因**：CSS 后处理器，支持现代 CSS 特性
- **核心优势**：
  - 自动添加浏览器前缀
  - 支持 CSS 模块化
  - 与 Tailwind CSS 深度集成
- **使用场景**：构建时的 CSS 转换

#### **@crxjs/vite-plugin v2**
- **选择原因**：Vite 的 Chrome 扩展构建插件
- **核心优势**：
  - 自动处理 manifest.json
  - 支持 HMR
  - 自动生成 background 和 content script 入口
- **使用场景**：浏览器扩展打包

## 3. 浏览器扩展技术选型

### 3.1 扩展规范

#### **Chrome Extension Manifest V3**
- **选择原因**：
  - 最新扩展标准，安全性更高
  - Google 强制要求，2022 年后新扩展必须使用
  - Chrome/Edge 市场份额最大
- **核心优势**：
  - 更好的权限控制
  - Service Worker 替代 background page
  - 内容安全策略（CSP）更严格
- **替代方案对比**：
  - **Manifest V2**：已弃用，2024 年后将无法上架
  - **Firefox WebExtension**：API 兼容性，但市场份额小
  - **Safari App Extension**：苹果生态，开发成本高

### 3.2 目标平台

#### **Chrome / Edge**
- **选择原因**：
  - Chromium 内核，扩展通用
  - Chrome 市场份额约 65%
  - Edge 市场份额约 5%
  - 开发一次，双平台发布
- **核心优势**：
  - 统一的扩展商店
  - 完善的开发工具
  - 活跃的开发者社区

## 4. 文件存储技术选型

### 4.1 本地文件操作

#### **File System Access API**
- **选择原因**：
  - 现代浏览器原生 API，无需额外权限
  - 支持目录选择和递归文件操作
  - 用户授权后可直接读写本地文件
- **技术规格**：
  - `showDirectoryPicker()`：目录选择器
  - `getFileHandle()`：获取文件句柄
  - `createWritable()`：创建可写流
- **核心优势**：
  - 无需传统下载方式
  - 支持大文件流式写入
  - 保持目录结构
- **替代方案对比**：
  - **传统 <a> 下载**：无法控制保存路径，文件散乱
  - **Chrome Downloads API**：需要额外权限，用户体验差
  - **NW.js/Electron**：需要打包桌面应用，违背扩展定位
- **使用场景**：
  - 用户选择导出目录
  - 创建子目录结构
  - 写入文档文件
  - 显示写入进度

### 4.2 配置存储

#### **localStorage**
- **选择原因**：浏览器原生 API，简单可靠
- **技术规格**：
  - 容量：约 5-10MB
  - 类型：仅支持字符串
  - 作用域：同源下共享
- **核心优势**：
  - 同步读写
  - 持久化存储
  - API 简单
- **使用场景**：
  - 导出格式配置
  - 用户偏好设置

## 5. 第三方服务选型

#### **Chrome Web Store**
- **选择原因**：全球最大的浏览器扩展商店

- **核心优势**：
  - 用户基数大

  - 评分和评论系统
- **替代方案对比**：
  - **Edge Add-ons**：市场份额小，但提交免审核费
  - **Firefox Add-ons**：市场份额小，审核更严格
  - **独立分发**：用户手动安装，更新困难
- **使用场景**：
  - 扩展发布
  - 版本更新
  - 用户反馈收集

#### **Edge Add-ons**
- **选择原因**：Edge 浏览器官方商店
- **技术规格**：
  - 审核周期：1-2 个工作日
  - 费用：免费
  - 覆盖：Edge 浏览器
- **核心优势**：
  - 免费提交
  - 审核速度快
  - 与 Chrome 扩展通用
- **使用场景**：Edge 浏览器用户扩展分发

### 5.2 钉钉开放服务

#### **钉钉文档 API**
- **选择原因**：钉钉官方提供的文档操作接口
- **技术规格**：
  - 基础域名：alidocs.dingtalk.com / docs.dingtalk.com
  - 认证方式：Cookie + A-Token
  - 限流：动态调整
- **核心优势**：
  - 官方支持，稳定可靠
  - 功能完整
  - 无需额外注册应用
- **使用场景**：
  - 获取文档内容
  - 创建导出任务
  - 下载转换后的文件

### 5.3 代码托管平台

#### **GitHub**

- **技术规格**：
  - 免费公开仓库
  - Issues 和 PR 支持

- **核心优势**：
  - 开源社区活跃

## 6. 开发工具选型

## 9. 技术选型总结

### 核心原则
1. **轻量优先**：选择轻量级技术栈，保持扩展体积小
2. **兼容老项目**：继承 ding-doc-downloader 成熟代码，降低风险
3. **渐进增强**：使用现代浏览器 API，提供降级方案
4. **用户隐私**：所有数据本地处理，不上传任何内容
5. **开箱即用**：无需注册账号，安装即用

### 技术亮点
✅ **自研框架**：400 行实现响应式和组件化，极致轻量
✅ **现代 API**：File System Access API 实现目录级文件操作
✅ **零成本架构**：无需服务器，纯前端实现
✅ **Apple 风格**：Tailwind CSS 定制，视觉体验出色
✅ **格式丰富**：支持 5+ 种导出格式，覆盖主流需求
✅ **成熟复用**：基于老项目代码，降低 50%+ 开发风险

### 升级路径

| 阶段 | 用户数 | 升级方案 |
|------|--------|---------|
| **初期** | < 1000 | 当前架构，按需修复 Bug |
| **成长期** | 1000-10000 | 考虑迁移到 Vue 3，提升可维护性 |
| **扩展期** | > 10000 | 添加后端服务，支持用户统计和付费功能 |

### 核心技术运维建议

**日常维护：**
- 监控钉钉 API 变化，及时适配
- 关注 Chrome/Edge 更新，测试兼容性
- 收集用户反馈，优化交互体验
- 定期更新依赖，修复安全漏洞

**升级时机：**
- 用户数突破 5000 时考虑框架迁移
- Chrome 强制政策变更时适配
- 钉钉 API 大版本更新时重构
- 性能问题严重影响体验时优化
