## 1. 总体技术架构

### 架构模式
- **架构类型**：CLI 命令行工具（单体应用）
- **部署方式**：npm 全局安装 / 本地开发模式
- **核心特性**：
  - 跨平台支持（Windows / macOS / Linux）
  - 智能Cookie管理（Playwright自动登录）
  - 增量导出支持
  - 并发控制（避免API限流）
- **模块化设计**：
  - CLI 入口层
  - 业务逻辑层
  - API 调用层
  - 数据转换层

## 2. 核心技术选型

### 2.1 运行时环境
- **Node.js >= 18.0.0**
  - **选择原因**：
    - CLI工具标准运行时
    - 生态丰富，HTTP客户端库成熟
    - 跨平台支持良好
  - **核心优势**：
    - npm 全局安装，开箱即用
    - 与飞书导出工具保持一致，便于代码复用
    - 异步 I/O 性能优异
  - **替代方案对比**：
    - Python 3.10+：
      - 优势：老项目有Python实现，可参考
      - 劣势：包管理不如npm便捷，用户安装门槛较高
    - Bun：
      - 优势：性能更高，兼容Node.js
      - 劣势：生态较新，稳定性待验证

### 2.2 编程语言
- **TypeScript 5.3+**
  - **选择原因**：
    - 编译时类型安全，减少运行时错误
    - 优秀的IDE支持和代码提示
    - 与飞书导出工具保持一致
  - **核心优势**：
    - 接口定义清晰，便于团队协作
    - 重构更安全
    - 生成类型定义文档（.d.ts）
  - **替代方案对比**：
    - JavaScript：
      - 优势：无需编译，直接运行
      - 劣势：缺乏类型检查，大型项目维护困难
    - 纯Python：
      - 优势：老项目可复用
      - 劣势：无法复用飞书工具代码

### 2.3 CLI 框架
- **Commander.js 11.x**
  - **选择原因**：
    - 最流行的 Node.js CLI 框架
    - GitHub 25k+ stars，社区成熟
    - 飞书导出工具使用相同框架
  - **核心优势**：
    - 自动生成帮助信息
    - 支持子命令和选项解析
    - 支持 Git 风格的子命令
  - **替代方案对比**：
    - Yargs：
      - 优势：功能更丰富
      - 劣势：API相对复杂，学习曲线较陡
    - Oclif：
      - 优势：功能强大，插件化
      - 劣势：过度设计，不适合简单CLI工具
    - CAC (Command And Control)：
      - 优势：轻量级
      - 劣势：社区较小

### 2.4 交互式配置
- **Inquirer.js 9.x**
  - **选择原因**：
    - 优雅的交互式命令行体验
    - 支持多种输入类型（输入、选择、确认等）
  - **核心优势**：
    - 美观的终端UI
    - 支持验证和默认值
    - 异步支持
  - **替代方案对比**：
    - Prompts：
      - 优势：更轻量
      - 劣势：功能相对简单
    - Enquirer：
      - 优势：插件化
      - 劣势：社区不如Inquirer活跃

### 2.5 进度显示
- **cli-progress 3.x**
  - **选择原因**：
    - 专为CLI设计的进度条库
    - 支持多种进度条样式
  - **核心优势**：
    - 实时显示导出进度
    - 支持多个进度条并行
    - 可自定义格式

## 3. 网络通信选型

### 3.1 HTTP 客户端
- **axios 1.x**
  - **选择原因**：
    - Promise API，使用简洁
    - 拦截器支持，便于统一处理
    - 自动转换JSON数据
  - **核心优势**：
    - 请求/响应拦截器
    - 超时配置
    - 取消请求支持
  - **替代方案对比**：
    - got：
      - 优势：更现代的API
      - 劣势：生态相对较小
    - node-fetch：
      - 优势：与Web Fetch API一致
      - 劣势：功能相对简单

### 3.2 Cookie 管理
- **tough-cookie 4.x**
  - **选择原因**：
    - RFC 6265 标准实现
    - 与 axios 集成良好
  - **核心优势**：
    - Cookie 持久化
    - 自动处理过期
    - 支持内存和文件存储

### 3.3 HTML 解析
- **cheerio 1.x**
  - **选择原因**：
    - 类 jQuery API，学习成本低
    - 轻量高效
  - **核心优势**：
    - 从钉钉HTML页面提取 dentryKey
    - 解析内嵌JSON数据
  - **替代方案对比**：
    - jsdom：
      - 优势：完整DOM实现
      - 劣势：重量级，性能较低
    - node-html-parser：
      - 优势：更快
      - 劣势：API不如cheerio友好

## 4. 自动化登录技术

### 4.1 浏览器自动化
- **Playwright 1.40+** ⭐核心技术亮点
  - **选择原因**：
    - 微软出品，维护活跃
    - 跨浏览器支持（Chromium、Firefox、WebKit）
    - 老项目已验证可行性
  - **技术规格**：
    - 无头模式（Headless）
    - 自动保存Cookie
    - 支持用户登录后自动关闭
  - **核心优势**：
    - 解决Cookie过期问题（7-30天有效期）
    - 无需用户手动配置Cookie
    - 支持扫码登录和账号密码登录
  - **替代方案对比**：
    - Puppeteer：
      - 优势：更早推出
      - 劣势：仅支持Chromium，API不如Playwright友好
    - Selenium：
      - 优势：支持更多浏览器
      - 劣势：配置复杂，性能较差
  - **使用场景**：
    - Cookie失效时自动重新登录
    - 用户首次使用时引导登录
    - 支持多账号切换

## 5. 工具库选型

### 5.1 日志管理
- **pino 8.x**
  - **选择原因**：
    - 高性能日志库
    - 结构化日志输出
  - **核心优势**：
    - JSON格式，便于后续分析
    - 支持日志级别（debug/info/warn/error）
    - pino-pretty 提供美观的终端输出
  - **替代方案对比**：
    - Winston：
      - 优势：功能丰富
      - 劣势：性能较差
    - Bunyan：
      - 优势：类似pino
      - 劣势：维护不够活跃

### 5.2 配置管理
- **cosmiconfig 8.x**
  - **选择原因**：
    - 支持多种配置文件格式
    - 被广泛使用（ESLint、Prettier等）
  - **核心优势**：
    - 支持 .rc、.json、.js、.cjs 等格式
    - 自动搜索配置文件
    - 支持配置扩展
  - **支持的配置文件**：
    - `.dingding-exportrc`
    - `.dingding-exportrc.json`
    - `.dingding-export.config.js`
    - `dingding-export` 字段在 `package.json`

### 5.3 并发控制
- **p-limit 3.x**
  - **选择原因**：
    - 轻量级并发控制库
    - Promise 友好
  - **核心优势**：
    - 控制同时执行的Promise数量
    - 避免触发钉钉API限流
    - 可配置并发数（默认5，最大20）

### 5.4 文件匹配
- **glob 10.x**
  - **选择原因**：
    - Node.js 标准glob库
  - **使用场景**：
    - 匹配批量导出的文档ID文件
    - 查找配置文件

### 5.5 路径处理
- **path-extra (或 Node.js 内置 path)**
  - **使用场景**：
    - 确保跨平台路径兼容
    - 处理相对路径和绝对路径

## 6. 开发工具选型

### 6.1 代码规范
- **ESLint + TypeScript ESLint**
  - **选择原因**：
    - 业界标准
    - TypeScript 支持
  - **配置**：
    - 继承 `@typescript-eslint/recommended`

- **Prettier 3.x**
  - **选择原因**：
    - 统一代码风格
    - 与 ESLint 集成良好

### 6.2 Git 钩子
- **Husky 8.x + lint-staged**
  - **选择原因**：
    - Git hooks 管理标准
    - 提交前自动检查
  - **配置**：
    - pre-commit: 运行 lint-staged
    - commit-msg: 验证提交信息格式

### 6.3 提交规范
- **Commitlint**
  - **选择原因**：
    - 强制统一的提交信息格式

  - **配置**：
    - 使用 Conventional Commits 规范
    - feat/fix/docs/style/refactor/test/chore 等类型

### 6.4 包管理
- **pnpm 8.x**
  - **选择原因**：
    - 节省磁盘空间
    - 安装速度快
    - 严格依赖管理
  - **替代方案**：
    - npm：
      - 优势：无需额外安装
      - 劣势：node_modules 占用空间大
    - yarn：
      - 优势：功能丰富
      - 劣势：存在幽灵依赖问题

## 11. 技术选型总结

### 核心原则
1. **与飞书导出工具保持一致**：便于代码复用和维护
2. **成熟稳定优先**：选择社区成熟的库，降低风险
3. **类型安全**：TypeScript 确保代码质量
4. **用户体验**：智能Cookie管理减少配置复杂度
5. **跨平台兼容**：确保 Windows/macOS/Linux 一致体验

### 技术亮点
✅ **智能 Cookie 管理**：Playwright 自动登录，解决过期问题
✅ **两阶段 API 调用**：先获取 dentryKey，再获取文档内容
✅ **并发控制**：p-limit 避免触发 API 限流
✅ **增量导出**：仅导出有更新的文档，节省时间
✅ **模块化设计**：清晰的分层架构，便于扩展
✅ **类型安全**：TypeScript 编译时检查

### 升级路径

| 阶段 | 用户规模 | 升级方案 |
|------|---------|---------|
| **初期** | < 100用户 | 当前架构足够 |
| **成长期** | 100-1000用户 | 增加错误上报（Sentry） |
| **扩展期** | > 1000用户 | 考虑服务端辅助（如进度同步） |

### 技术栈运维建议

**日常维护：**
- 定期更新依赖包（每月一次）
- 监控钉钉API变化
- 收集用户反馈优化功能
- 维护文档和示例

**升级时机：**
- 钉钉API发生重大变更
- Node.js LTS 版本更新
- 安全漏洞修复
- 用户量激增需要优化性能
